## 📋 完成的需求清单

### 0311 更新需求（✅ 全部实现）

#### 1. 时间档位优化 ⏰

**需求**：活动具体时间从半小时一档改为一小时一档，便于借教室时内部不冲突

**实现方式**：
- **后端**：在 `.env` 新增 `TIME_SLOTS` 配置项
  ```env
  TIME_SLOTS=09:00,10:00,11:00,12:00,13:00,14:00,15:00,16:00,17:00,18:00,19:00,20:00,21:00,22:00
  ```
- **数据库**：`活动时间` 字段存储如 "10:00-11:00" 的格式
- **前端**：显示选中的时间，及时反馈用户

**益处**：
- 一小时的固定时长便于教室管理
- 易于检测时间冲突
- CAC可以更好地安排场地

#### 2. 申请旁听登记选项 👂

**需求**：增加申请旁听登记选项，便于统计借教室人数

**实现方式**：
- **数据库设计**：在"分享会报名"表中新增 `角色` 字段（单选：评议员/旁听）
- **后端API**：`POST /api/signup` 支持接收 `role` 参数
- **前端UI**：美化的角色选择按钮组，用户体验更直观

**代码示例**（前端）：
```html
<label>我的角色
  <div class="role-group">
    <input type="radio" id="role-reviewer" name="role" value="评议员">
    <label for="role-reviewer">👨‍⚖️ 评议员</label>
    
    <input type="radio" id="role-listener" name="role" value="旁听">
    <label for="role-listener">👂 旁听</label>
  </div>
</label>
```

**益处**：
- 明确区分评议员和旁听者角色
- 易于统计教室需要容纳的人数
- 为未来的参与度分析奠定基础

#### 3. 自动计算拟参加人数 📊

**需求**：增加拟参加人数（自动计算，分享人数+旁听人数+3名评议员）

**实现方式**：
- **后端函数** (`app.py` 第 131 行）：
  ```python
  def _calculate_expected_attendance(activity_id):
      """计算拟参加人数 = 分享者数 + 旁听者数 + 3名评议员"""
      speakers_count = len(speakers_list)  # 从活动表读取
      listener_count = _count_signups_by_activity(activity_id, '旁听')
      reviewer_count = REVIEWER_LIMIT  # 固定3人
      return speakers_count + listener_count + reviewer_count
  ```
- **前端展示** (`index.html` 第 444 行）：
  ```javascript
  拟参加人数：${stats.expected_attendance || 0}
  // 分享 + 旁听 + 评议
  ```
- **API端点**：`GET /api/activities` 返回计算好的数据

**益处**：
- 主持人可快速了解现场人数，做好场地准备
- 自动计算，减少人工错误
- 实时更新，反映报名情况变化

### 隐患改进方案（✅ 第一期已实现）

#### 隐患 #5 & #9 & #10：技术统计机制

**当前隐患**：
- #5 延迟症导致没有意识到自己突破社团边界
- #9 缺少技术统计手段，无法及时发现快要突破边界的社员
- #10 缺少技术统计手段，无法及时发现快要消亡的兴趣组

**改进方案**（已实现）：

1. **越界社员监控** 🔴
   - **功能**：自动检测参与次数超过阈值的社员
   - **实现** (`app.py` 第 266 行）：
     ```python
     def _detect_boundary_violations():
         """检测接近超出边界的社员（过度参与）"""
         # 阈值：超过10次报名
         violations = {name: count for name, count in member_participation.items() 
                       if count > warning_threshold}
         return violations
     ```
   - **前端展示**：在页面下方的预警橱窗显示
     ```html
     <div class="warnings" style="background: #fecaca;">
       <strong>⚠️ 参与度过高警告</strong><br/>
       张三: 12 次 → 建议休息调整
     </div>
     ```
   - **数据来源**：分析"分享会报名"表中该成员的条目数

2. **消亡兴趣组检测** 🟡
   - **功能**：统计活跃分享者，发现无活动的兴趣组
   - **实现** (`app.py` 第 244 行）：
     ```python
     def _detect_inactive_members():
         """检测消亡的兴趣组"""
         speakers_set = set()
         for activity in activities:
             speakers = [s.strip() for s in speakers_str.split(',') if s.strip()]
             speakers_set.update(speakers)
         return {
             'total_active_speakers': len(speakers_set),
             'message': f'当前有 {len(speakers_set)} 个活跃的分享者/兴趣组'
         }
     ```
   - **前端展示**：
     ```html
     <div class="warnings">
       📊 当前有 5 个活跃分享者/兴趣组
       (说明：还有多少兴趣组需要激励)
     </div>
     ```
   - **数据来源**：分析"分享会活动"表中的分享者字段

3. **API 端点** (`app.py` 第 373 行）：
   ```python
   @app.get("/api/stats")
   def api_stats():
       """获取全局统计和预警信息"""
       return {
           'inactive_groups': {...},
           'boundary_violations': {...}
       }
   ```

**益处**：
- ✅ CAC 可快速发现问题
- ✅ 数据驱动的管理决策
- ✅ 保护社员：及时发现过度参与者
- ✅ 保护社团：及时发现消亡兴趣组

- ✅ 关注点分离（What vs Who）

## 💡 设计亮点

### 1. **两表分离设计**
   - 活动是独立的资源，多次使用
   - 报名是事件，一次一条记录
   - 解耦清晰，易于维护

### 2. **灵活的统计回调**
   - 统计函数独立，易于测试
   - 可轻松添加新的统计维度
   - API直接暴露，便于第三方集成

### 3. **前端智能UI**
   - 根据数据动态生成表单
   - 根据状态动态禁用选项
   - 实时反馈用户操作

### 4. **清晰的错误提示**
   - 区分不同错误类型（验证、业务、系统）
   - 用户友好的错误信息
   - 便于排查问题
